Three new views, two explorations, zero changes to the data layer.

The v2 prototype proposes three new views — a chronograph dial, a live oscilloscope, and a request waterfall — all powered by the Svelte stores that already exist. Two further views (Strata for long-range distributions, Terminal for a forensic event log) are proposed on paper and need prototyping before they ship. This package maps every piece of the prototype to concrete files, types, tokens, and a phased rollout.

02 Views

Three prototyped views plus an Enriched upgrade, two proposed explorations, one legacy kept for safety.

Three views (Overview, Live, Atlas) are in the prototype and ready to implement. Overview ships in two phases — Classic first, then an Enriched upgrade that layers on a baseline arc, quality trace, racing strip, event feed, and causal verdict. Two further views (Strata, Terminal) are design direction only — prototype them before building. The legacy Lanes view ships one more release as an escape hatch.

06 Rollout

Seven phases plus a 2.5 sub-phase. Each ships independently.

Nothing in a later phase is a prerequisite for shipping an earlier one. Phase 0 lands the plumbing before any visual work; phases 5–6 require design prototyping before they can be scheduled; phase 7 retires the legacy views once users are on the new defaults.

PHASE 0

Foundations

Tokens, classifier, format helpers, type additions, persistence migration. No visual change.

1–2 days
PHASE 1

Rail + view switcher

Persistent 264px left rail, seven-way topbar switcher, ViewSwitcher routing. Stubs for new views.

1 day
PHASE 2

Overview · Classic

Chronograph dial with orbit ring, live hand interpolation, threshold arc, verdict strip, metrics triptych. Ship as the default before Enriched lands.

2–3 days
PHASE 2.5

Overview · Enriched layers on 2

Dial v2 (baseline arc, quality trace, within-band label, breathing chrome). Add RacingStrip, EventFeed, CausalVerdictStrip. Extract computeCausalVerdict() to utils/verdict.ts. Flip default once soaked.

3–4 days
PHASE 3

Live

ScopeCanvas (reuse timeline-renderer.ts core), unified/split toggle, solo focus mode, tooltips.

2–3 days
PHASE 4

Atlas

Extract PhaseBar from lane header. Waterfall with P50/P95 toggle, hypothesis strip, sample strip.

2 days
PHASE 5

Strata prototype first

Prototype the view, get sign-off, then build: per-row percentile bands (ticks, not KDE in v1), time range selector, low-confidence indicator.

2 days after prototype
PHASE 6

Terminal prototype first

Prototype the view, get sign-off, then build: event derivation, severity chips, filter pills, JSONL export. Gated behind showAdvancedViews.

2 days after prototype
PHASE 7

Deprecation

Retire timeline, heatmap, split. Bump persisted version. Escape hatch via URL.

1 day
Total estimate: ~11–14 engineer-days for the three prototyped views (Overview Classic + Enriched, Live, Atlas + foundations & rail). Strata and Terminal add ~2 days each on top, but only after each is prototyped and signed off. No data-layer work is required — the prototype’s synthetic generator is deleted; every piece of data the views display is already in the existing Svelte stores.

Design decisions

Locked before handoff.

These were explored during prototyping and decided. Re-opening any of them is allowed but should be an explicit call, not an accident in implementation.

Dial style
Minimal only. Mechanical (brushed bezel, jewel screws, sub-dial) explored and rejected — too ornamental for a monitoring surface.
Scope style
Clean only. CRT (scanlines, vignette, phosphor glow) explored and rejected — reduced legibility, skeuomorphic for skeuomorphism’s sake.
Endpoint focus
Global, driven by the rail. No per-view endpoint pickers. One source of truth: uiStore.focusedEndpointId.
Tier-2 palette
Existing token palette, reused everywhere. Waterfall, sample strip, hypothesis evidence all use the same five colors. No phase colors invented.
Rail content
No sparklines. The rail is chrome; every view has its own primary visualization. Pip glow is the only quiet live-motion signal.
Threshold UX
Read-only on views. Threshold changes through Settings drawer only — prevents accidental edits while reading a trace.
Out of scope: Settings drawer redesign, Share popover redesign, mobile/tablet layouts, light mode. All deferred to a later cycle.
